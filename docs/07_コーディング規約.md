# 07 コーディング規約

## 目的

- 本プロジェクトにおける実装方針、命名方針、レイヤー構成を統一する
- 変更時の判断基準を明確にし、保守しやすいコードベースを維持する

## 共通方針

- 実在企業名、社内用語、機密情報をコミットしない
- 仕様変更時は実装とあわせて `docs` を更新する
- 画面文言は日本語ベース、識別子は英語ベースで統一する
- API の入出力は DTO を通して定義する
- 責務を分離し、Controller に業務ロジックや SQL を直接書かない

## ブランチ運用

- `main`: 安定版
- `develop`: 開発統合
- `feature/*`: 機能開発
- `fix/*`: 不具合修正

## フロントエンド方針

- Vue は Composition API を採用する
- 画面表示用の責務と API 通信の責務を分離する
- モック表示で終わらせず、API 接続時は通信失敗時の考慮も入れる

## バックエンド方針

- Backend は `controller -> service -> dao -> mapper` の順で責務を分離する
- `controller` は HTTP 入出力の制御のみを担当する
- `service` は業務ロジックを担当し、`*Service` interface と `*ServiceImpl` 実装クラスを対で配置する
- `dao` はデータアクセスの窓口とし、`*Dao` と `*DaoImpl` を対で配置する
- `mapper` は DB 接続用インターフェースとし、MyBatis で SQL と接続する
- `model` はドメインモデルを配置する

## MyBatis 構成ルール

- `backend/src/main/java/com/projectmanagementapp/domain` 配下に `dao`、`model`、`service` を置く
- DB 接続用の Mapper interface は `backend/src/main/java/com/projectmanagementapp/domain/dao/mapper` 配下に置く
- SQL の実体は `backend/src/main/resources/mapper` 配下の `*Mapper.xml` に置く
- Mapper interface に SQL を直接書かない
- Mapper XML の `namespace` は対応する Java の Mapper interface と一致させる
- `controller` から mapper を直接呼ばず、必ず `service -> dao -> mapper` の順で呼ぶ
- 新しい DB アクセス処理を追加する場合も、この構成と命名規則を維持する

## 命名と実装ルール

- クラス名、メソッド名、変数名は役割が分かる名前にする
- DTO、DAO、Mapper、Model は用途ごとに明確に分ける
- `service` 層も interface と実装クラスを分け、Controller からは interface を参照する
- SQL を追加・変更する場合は、Java 側の mapper と XML 側の定義をセットで更新する
- 例外は握りつぶさず、必要に応じて適切な HTTP ステータスやメッセージへ変換する
- 不要な重複実装を避け、共通化できる処理は service などで整理する
